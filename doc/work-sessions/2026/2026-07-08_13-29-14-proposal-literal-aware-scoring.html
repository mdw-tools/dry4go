Literal-Aware Duplication Scoring

Draft — awaiting review 2026-07-08 mdw-tools/dry4go Proposal (Picard phase 2)

1 · Background

Running dry4go on its own repository reports this pair at a perfect score:

DUPLICATE score=1.00
  internal/dry/dry_test.go:64-101   TestFindDuplicatesMatchesMethodsAndCompositeLiterals
  internal/dry/dry_test.go:103-114  TestFiltersShortFunctions

These two tests are semantically opposite — one asserts that a duplicate is found, the other that one is not — yet they score 1.00 because the current normalizer erases every identifier and every literal value. What remains of each test is the identical skeleton: write two files, call FindDuplicates, check the error, check the candidate count. All of the meaning lives in literals (the Go source strings under test, the thresholds, the expected counts), and all of it is discarded before scoring.

Goal. Make the score reflect literal content proportionally: a pair that is identical except for one small constant should still score ≈1.0 (that is real, parameterizable duplication), while a pair whose shared structure is mere scaffolding around large, divergent literals — the common shape of test functions — should fall below the reporting threshold.

2 · How scoring works today

The two erasure rules that matter here (dry.go:392-398):

case *ast.Ident:      // every identifier becomes the same token
    return node{Tag: "ident"}
case *ast.BasicLit:   // "left.go", 0.75, and 42 all become literal/<kind>
    return node{Tag: "literal/" + x.Kind.String()}

Erasing identifiers is correct and should not change — renamed variables are the classic disguise of real duplication. Erasing literal kinds-only is the problem: it treats "package sample\nfunc A…" and "package sample\ntype Invoice…" as interchangeable.

3 · Proposed approach — weighted Jaccard with literal features

Keep the structural fingerprint set exactly as it is, and add a second family of features: one feature per distinct literal value in the function, carrying a weight that grows with the literal's textual length. Similarity becomes a single weighted Jaccard over the union of both families:

Features per function

  • Structural: every serialized subtree, weight 1 each (unchanged from today).
  • Literal: lit/<kind>/<value> for each distinct *ast.BasicLit, weight 1 + min(len(value), 56) / 8 — i.e. in the range [1, 8].

Score

score = Σ min(wL(f), wR(f)) / Σ max(wL(f), wR(f))  over all features f present on either side.

Why this shape rather than the alternatives (§5):

  • Self-scaling. No global blend knob to tune. Literal influence is proportional to how much of the function is literals. A 100-node function that differs in one small int barely moves (≈0.98); a test whose body is two 30-line string literals is dominated by them, exactly as a human reader would judge it.
  • Non-brittle. Because literal values stay out of the structural tree, changing one constant does not invalidate every ancestor subtree fingerprint. Each divergent literal costs only its own weight.
  • Length-weighted, capped. A multi-line Go-source string is stronger evidence of dissimilarity than a differing 0 vs 1, but the cap (weight 8) keeps a single enormous string from single-handedly vetoing an otherwise-identical pair.
  • Perfect duplicates still score 1.00. Identical code shares all features of both families.

Edge rules

  • Functions with no literals on either side: literal family contributes nothing; score reduces to today's structural Jaccard. No special case needed — it falls out of the formula.
  • true/false/nil already carry their value in the structural tag (literal/true …); they get no literal feature.
  • Set semantics (distinct values), matching the existing structural fingerprint set. Repetition counts are ignored; see trade-offs.

Projected outcomes (estimates, to be confirmed by tests)

Pair Score today Projected Reported at 0.82?
Byte-identical functions 1.00 1.00 yes — unchanged
Real dup, renamed identifiers, same literals 1.00 1.00 yes — unchanged
Real dup differing in one small constant 1.00 ≈0.97 yes
The offending test pair (scaffolding + big divergent strings) 1.00 ≈0.55–0.65 recategorized as SHAPE-TWIN (§4)
Dup differing only in a few error-message strings 1.00 ≈0.85–0.92 yes (dips with string mass)

Implementation shape

  • entry gains literals map[string]float64 (feature → weight), collected in scanFile by a small ast.Inspect walk over each function — the normalizer itself is untouched.
  • similarity produces two numbers per pair: the structural Jaccard (today's score, unchanged) and the combined weighted Jaccard described above.
  • Candidate gains Kind (duplicate | shape-twin) and StructuralScore (JSON: kind, structural_score); classification happens in FindDuplicates and rendering in both formatters (§4).
  • No new CLI options; no change to Options or defaults.

4 · Shape twins — preserving the table-driven-test signal

The literal-aware score resolves the motivating pair without any test-file special-casing: scaffolding-shaped tests differ almost entirely in literals, so their combined score falls below threshold naturally, while genuinely copy-pasted test logic (same fixtures, same assertions) still surfaces as a duplicate.

But "same skeleton, different data" is itself a useful finding — it is the classic precondition for a table-driven test, or for parameterizing production code. Silently dropping those pairs would trade one blind spot for another. Since both scores are already computed, the tool keeps such pairs visible under a distinct label instead:

DUPLICATE score=0.97
  internal/dry/dry.go:453-459
  internal/dry/dry.go:461-467

SHAPE-TWIN structural=1.00 score=0.61 (consider table/parameterization)
  internal/dry/dry_test.go:64-101
  internal/dry/dry_test.go:103-114
Classification rule. A pair is a DUPLICATE when its combined score ≥ threshold. Otherwise it is a SHAPE-TWIN when its structural score ≥ 0.95 (internal constant) — near-identical shape whose divergence is essentially all literal. Anything else is not reported. Duplicates sort before shape twins in the output; JSON candidates carry kind and structural_score so downstream tooling can filter either way.

This reframes the original complaint precisely: the offense was never that the test pair got mentioned — it was the claim of a perfect 1.00 duplicate. As a shape twin it is honestly described, and the reader can judge (as we did) whether two tests justify a table. A --skip-tests flag was considered for teams that never want test findings, and dropped in review as too blunt.

5 · Alternatives considered

Alternative Why not
Put literal values into the structural tag (literal/STRING/"left.go") Brittle in the opposite direction: one changed constant alters every ancestor subtree fingerprint, so a real dup differing in a single number would crater to a very low score. Loses the "parameterizable duplication" signal entirely.
Two separate scores blended by a constant (0.8·structural + 0.2·literal) Adds a tuning knob with no principled setting, and the literal channel gets the same influence whether the function is 5% or 95% literals. Weighted Jaccard achieves the blend with the proportionality built in.
Skip _test.go files (flag or default) Blunt: copy-pasted test logic is real duplication worth finding, and the underlying scoring flaw would still affect literal-heavy production code (SQL, templates, table-driven data). Considered as an opt-in flag in an earlier draft; dropped in review.
Penalty multiplier when literal sets diverge Equivalent in spirit but needs both a penalty curve and a mass estimate as separate tunables; weighted Jaccard subsumes both.

6 · Trade-offs & open questions

  • Two magic constants remain (length cap 56, divisor 8). They are local to the weight function and validated by the projected-outcomes tests, but they are judgment calls. If tests show borderline behavior, tune here first.
  • Textual, not semantic, equality. 0x10 vs 16, or "a" + "b" vs "ab", count as divergent literals. Acceptable: textual divergence in a true dup is rare and costs little weight.
  • Set semantics ignore repetition. A literal used once vs ten times is one feature. Consistent with the existing structural fingerprint set; multiset weighting is a possible future refinement if it proves to matter.
  • Scores generally drop, never rise. Any pair sharing all structure and all literals is unaffected; every other pair scores the same or lower. Users with pinned custom thresholds may see fewer findings — this is the intended direction, but worth a release note.
  • Existing test expectations may shift. The current test corpus asserts candidate counts at specific thresholds; each assertion must be re-verified (and is called out in the checklist).
